# SEO Tools Directory

Nuxt app for a SEO-friendly tools directory. The app renders one detail page for each record in `data/data.json`, links all records from the index page, and generates SEO metadata, sitemap, robots, and JSON-LD structured data.

## How To Run

Install dependencies:

```bash
npm install
```

Start the development server:

```bash
npm run dev
```

Open:

```txt
http://localhost:3000
```

Build and preview the SSR production app:

```bash
npm run build
npm run preview
```

Generate and preview the static SSG output:

```bash
npm run generate
npm run preview
```

## SSR And SSG

This project keeps Nuxt SSR enabled:

```ts
ssr: true
```

That is important for SEO because Nuxt can render real HTML for crawlers instead of relying on a client-only JavaScript app.

For this directory, most public pages are good candidates for SSG because the content comes from local static data:

- `/`
- `/tools/[slug]`

The routes are prerendered in `nuxt.config.ts`:

```ts
nitro: {
  prerender: {
    routes: ["/", ...data.map((item: any) => `/tools/${item.slug}`)],
  },
}
```

Use SSG when the directory data does not need to change on every request. Use SSR when a page must fetch fresh live data for each visitor, such as user-specific content, real-time API data, or request-based content.

In short:

```txt
npm run dev       = local development
npm run generate  = static SEO-friendly SSG output
npm run build     = SSR production build
npm run preview   = preview generated/build output
```

## SEO Coverage

Each tool detail page provides SEO data from the tool record:

- unique `<title>`
- meta description
- canonical URL
- Open Graph title, description, URL, and image
- one `<h1>`
- JSON-LD structured data

The index page links to every detail page using each tool slug.

Sitemap and robots are generated by:

```ts
modules: ["@nuxtjs/sitemap", "@nuxtjs/robots"]
```

Before deploying, update the production URL in `nuxt.config.ts` and the detail page `siteUrl` value from local/dummy URLs to the real domain.

## Schema Type

The detail pages use JSON-LD with:

```json
{
  "@type": "SoftwareApplication"
}
```

`SoftwareApplication` is a good fit because each record represents a software/tool listing with fields such as:

- name
- category
- description
- image
- pricing
- rating
- review count
- website URL

The schema currently includes `Offer` for pricing and `AggregateRating` for rating/review data. If the page later becomes a long editorial review, an additional `Review` or `Article` schema can be added, but the core tool entity should remain `SoftwareApplication`.

## Data Source

Tool records live in:

```txt
data/data.json
```

Each record should include at least:

```json
{
  "id": 1,
  "slug": "bioglow",
  "name": "BioGlow",
  "category": "Link in Bio",
  "tagline": "Turn a single profile link into a mini landing page.",
  "description": "BioGlow is a link in bio tool built for creators and small brands.",
  "features": ["custom domains", "unlimited links", "click analytics"],
  "pricing": "$9/mo",
  "platforms": ["Web", "Android"],
  "rating": 4.7,
  "reviews": 1544,
  "website": "https://example.com/tools/bioglow",
  "image": "https://picsum.photos/seed/bioglow/600/400"
}
```
